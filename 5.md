# 5. Background of Payment Hub EE

## 5.1 Role of Payment Hub EE in the Mifos Ecosystem

Payment Hub EE is a modular payment orchestration platform developed within the Mifos ecosystem. It enables interoperability between financial institutions, mobile money providers, payment networks, and core banking systems by providing a connector-based integration layer. The platform plays a central role in Digital Public Infrastructure (DPI) initiatives, where it powers real-time payment processing for financial inclusion use cases and is incorporated into the Mifos Gazelle platform for demonstrating production-grade payment integrations.

Rather than requiring direct point-to-point integrations between financial systems, Payment Hub EE acts as middleware that abstracts external payment networks behind a unified connector interface. Each connector handles the protocol-specific details of communicating with systems such as the Mojaloop interoperability switch, Fineract core banking backends, GSMA Mobile Money APIs, and cloud storage providers. This design allows new payment systems to be integrated without modifying the core platform, making Payment Hub EE extensible across different financial ecosystems and regulatory environments.

---

## 5.2 High-Level Architecture of Payment Hub EE

Payment Hub EE is built as a distributed microservices architecture where each service (connector) is a standalone Spring Boot application communicating through Apache Camel integration routes and Zeebe BPMN workflow orchestration. The architecture can be understood in terms of five logical layers: external payment systems, payment hub connectors, a shared connector library, a workflow orchestration and messaging layer, and core financial systems.

At the top layer, external systems such as the Mojaloop payment switch, mobile money APIs, and banking institutions send and receive payment requests. These requests are handled by connector applications — each connector is a Spring Boot service with Apache Camel routes that transform, validate, and forward messages between external systems and the internal workflow engine. Zeebe, a horizontally scalable BPMN workflow engine, orchestrates multi-step payment processes (fund blocking, transfer execution, callback notifications) by dispatching jobs to connector-registered workers. The connectors register Zeebe job workers that execute specific steps within payment workflows, such as `block-funds`, `book-funds`, `release-block` (in the AMS connector), or `initTransfer`, `sendToMojaloop`, `transactionCallback` (in the channel connector).

The technology stack underlying this architecture includes Java as the primary language, Spring Boot as the application framework, Apache Camel for enterprise integration pattern (EIP) routing and message transformation, Zeebe (Camunda) for BPMN-based workflow orchestration with gRPC communication, and Gradle as the build system. Additional infrastructure components include Redis for caching (channel connector), Apache Kafka for event streaming (bulk processor), and AWS S3 / Azure Blob Storage for batch file management. All connectors share a common library (`ph-ee-connector-common`) that provides reusable DTOs, HTTP interceptors, Camel route base classes, cryptographic utilities, and Zeebe variable constants.

**Layer descriptions:**

- **External Payment Systems:** The external entities that initiate or receive payment transactions — Mojaloop interoperability switch, GSMA Mobile Money providers, and banking APIs.
- **Payment Hub EE Connectors:** Individual Spring Boot microservices, each responsible for integrating with a specific external system. Each connector registers Zeebe job workers and defines Apache Camel routes for message processing.
- **Shared Connector Library (`ph-ee-connector-common`):** A non-runnable library JAR consumed by all connectors. Provides shared DTOs, Camel route base classes (`ErrorHandlerRouteBuilder`), HTTP interceptor framework (`@EnableJsonWebSignature`), cryptographic utilities, and Zeebe workflow variable constants.
- **Workflow Orchestration & Messaging:** Apache Camel handles message routing, transformation, and error handling within each connector. Zeebe provides BPMN-based orchestration across connectors — dispatching jobs to workers for multi-step payment processes (e.g., `block-funds` → `book-funds` → `release-block`).
- **Core Financial Systems:** The backend financial infrastructure including Mifos/Fineract core banking, settlement networks, and clearing systems.

## 5.3 Connector Architecture

Payment Hub EE uses a modular connector architecture where each connector is an independent Spring Boot application that integrates with a specific external system or handles a specific payment processing concern. The `ph-ee-connector-channel` serves as the REST API gateway that receives payment initiation requests from external clients and forwards them to the Mojaloop switch and GSMA Mobile Money providers. The `ph-ee-connector-ams-mifos` integrates with the Fineract core banking platform to execute account-level operations such as fund blocking, fund booking, and fund release during payment processing. The `ph-ee-bulk-processor` handles batch payment file ingestion, splitting, formatting, and orchestration through Kafka event streaming and cloud storage backends (AWS S3, Azure Blob).

Each connector follows a consistent internal pattern: incoming requests are received via REST endpoints or Zeebe job worker callbacks, processed through Apache Camel integration routes that handle message transformation, validation, and error handling, and then forwarded to external systems or back to the Zeebe workflow engine. All connectors extend a shared Camel route builder class (`ErrorHandlerRouteBuilder`) from the connector-common library, which provides standardized error handling, authentication, and request interception across the ecosystem. Connectors also register Zeebe workers that execute specific steps within BPMN payment workflows — for example, the AMS connector registers workers for `block-funds`, `book-funds`, and `release-block`, while the channel connector registers 11 distinct worker types for transaction initialization, Mojaloop communication, and callback processing.

This architecture provides several key benefits. Modularity allows each connector to be developed, deployed, and scaled independently. Extensibility means that integrating a new payment system requires only creating a new connector application with the appropriate Camel routes and Zeebe workers, without modifying the core platform or existing connectors. The shared connector-common library eliminates code duplication by centralizing DTOs, interceptors, and route base classes, while the Zeebe workflow engine decouples the business process logic from the connector implementation details. This separation of concerns makes the platform adaptable to diverse financial ecosystems and regulatory environments.

## 5.4 Repository and Dependency Structure

Payment Hub EE is implemented as a multi-repository ecosystem with 40+ repositories under the `openMF` GitHub organization. Each repository is a standalone Gradle project with its own `build.gradle`, dependency declarations, and Gradle wrapper configuration. The core repositories that form the foundation of the platform include:

- **`ph-ee-connector-common`** — the shared foundation library (~187 Java files) providing DTOs, Camel route builders, HTTP interceptors, cryptographic utilities, and Zeebe variable constants. This is a library JAR, not a runnable application, and is consumed by virtually every other connector in the ecosystem.
- **`ph-ee-connector-ams-mifos`** — the Fineract AMS integration connector, with Camel routes for interoperability API calls, Zeebe workers for fund management workflows, and CXF-based REST clients for Fineract backend communication.
- **`ph-ee-connector-channel`** — the REST API gateway for payment initiation, supporting Mojaloop and GSMA Mobile Money protocols, with Swagger-generated GSMA stub APIs (~55 files), Redis caching, and 11 Zeebe worker types.
- **`ph-ee-bulk-processor`** — the batch payment processing engine, integrating AWS S3, Azure Blob Storage, Apache Kafka, and Cucumber BDD testing for bulk transaction file workflows.
- **`ph-ee-operations-app`** — the operations dashboard and monitoring application.

The connector-common library occupies a critical position in the dependency hierarchy: every connector depends on it for shared types, interceptors, and route base classes. The channel connector extends `ErrorHandlerRouteBuilder` from connector-common via class inheritance — the tightest coupling point in the ecosystem. The bulk processor enables connector-common's JWS interceptor framework via the `@EnableJsonWebSignature` annotation, which triggers a `@ComponentScan` and `@Import` of the interceptor package. The AMS connector consumes shared DTOs with validation annotations. These integration patterns mean that any change to connector-common's namespace or API surface cascades to all downstream consumers.

Currently, each repository independently declares and manages its own dependency versions within its `build.gradle` file. There is no centralized mechanism governing which versions of Spring Boot, Apache Camel, Zeebe, Netty, or other shared dependencies are used across the ecosystem. This has led to observable version drift — for example, the channel connector uses Camel 3.12.0 while the AMS connector and bulk processor use Camel 3.4.0, Lombok versions vary between 1.18.22 and 1.18.24 across repos (with scope mismatches between `implementation` and `annotationProcessor`), and the bulk processor carries duplicate `commons-io` versions (1.3.2 and 2.11.0) on the same classpath. This per-repository dependency management approach makes ecosystem-wide upgrades difficult to coordinate and introduces risks of transitive version conflicts, particularly in the Netty/gRPC/Zeebe dependency chain where version alignment is critical for runtime stability.
